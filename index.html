Agent Mode vs Composer Mode

Agent Mode

  • Best for complex, multi-step tasks requiring reasoning
  • Can browse files, understand project structure
  • Makes decisions about what files to modify
  • Handles tasks like "refactor the authentication system"
  • More autonomous - you give high-level instructions

Composer Mode (Cmd+I)

  • Better for specific, targeted edits
  • You select code first, then give instructions
  • More direct control over what gets changed
  • Good for "fix this function" or "optimize this component"
  • Faster for simple, focused tasks

Agent Operational Settings

üîÑ Always

Full-auto autonomous mode

  • AI agent immediately acts on your instructions‚Äîno prompts asking for approval
  • Uses full developer capabilities: modifying files, running commands, refactoring
  • Think of it as "set it and forget it"‚ÄîCursor takes over until task's done
  • You can disable "Auto-run" or add files/commands to a deny list if too aggressive

üß© Auto

Conservative mode with checks

  • Cursor suggests changes first, then waits for your approval before applying
  • Often the default for users on paid accounts
  • Gives autonomy but with safety checks
  • Note: Some users report it might act more like Ask mode after model updates

üõ†Ô∏è Ask / Manual

Chat assistant mode

  • Cursor behaves more like a chat assistant
  • Doesn't apply changes or run commands autonomously
  • You get suggestions and can use "Apply" buttons or pick manually
  • Maximum control over every edit or command

When to Use Each Mode

üîÑ Always

  • When you trust the agent to act on its own
  • For full automation of complex tasks
  • Building new features that touch multiple files
  • Refactoring large parts of your codebase

üß© Auto

  • When you want suggestions plus control
  • General development work
  • Medium-sized projects
  • Cursor still asks you first before applying

üõ†Ô∏è Ask/Manual

  • When you prefer to control every edit or command
  • Making specific, isolated changes
  • Working on performance-critical code
  • Want to minimize token usage/costs

Pro Tip

TL;DR: Use Always when you trust the agent to act on its own‚Äîfor full automation. Use Auto if you want suggestions plus control (Cursor still asks you first). Use Ask/Manual if you prefer to control every edit or command. Start with Smart Auto-Attach for most projects and adjust based on your specific needs.

GitHub & Version Control for Beginners

Why Version Control Matters

Version control is like a time machine for your code. It tracks changes, lets you collaborate safely, and saves you from losing work. GitHub is the most popular platform for hosting Git repositories.

üöÄ Getting Started: Browser-Based Workflow

New to GitHub? Start here! This guide focuses on using GitHub entirely in your browser‚Äîno command line required.

1

Logging In

First, let's get you logged in:

  1. Connect to your company VPN (if required)
  2. Go to github.com/login
  3. Enter your username (usually firstname.lastname_company)
  4. You'll be redirected to your company's SSO login
  5. Enter your normal company credentials

After that, you're in your organization's GitHub!

2

What is a Repository?

Think of a repository as a smart project folder:

  • It holds your files and folders
  • It remembers every version of your work
  • It lets you collaborate with others safely
  • It's like a shared filing cabinet that never forgets
3

Creating Your First Repository

When you're logged in:

  1. Click the green New Repository button
  2. Choose the Owner (you or your organization)
  3. Name your repo‚Äîkeep it simple like my-first-project
  4. Add a short description (optional)
  5. Set it to Private for internal work
  6. Check Add a README so it's not empty
  7. Click Create repository

Done! You now have your first repository.

4

Uploading Files in the Browser

No command line needed! Here's how to upload existing files:

  1. Go into your repository
  2. Click Add file ‚Üí Upload files
  3. Drag and drop your files or click to browse
  4. Add a commit message like "Add initial files"
  5. Click Commit changes

That's it! Your files are now saved and versioned on GitHub.

5

Creating Folders

GitHub doesn't allow empty folders. To create one:

  1. Click Add file ‚Üí Create new file
  2. In the filename box, type foldername/README.md
  3. Add some content to the README file
  4. Click Commit changes

GitHub will create the folder and put the README file inside it.

6

What is Cloning? (Optional)

So far, we've used GitHub entirely in the browser. But sometimes you want a copy on your computer:

  • Cloning downloads the repository to your machine
  • Click the green Code button in any repo
  • Copy the URL and run git clone [URL] in terminal
  • This lets you work offline and use local tools

For most beginners, browser uploads are perfect!

7

What is Branching? (Optional)

Branching creates a safe copy of your repository:

  • The main version is called main or master
  • A branch is like a sandbox for trying changes
  • When you're done, you can merge your branch back into main
  • Example: Create a docs-update branch to safely edit documentation

For simple uploads, you can stay on the main branch.

üéØ Quick Summary

  • ‚úÖ Connect to VPN, log in with company SSO
  • ‚úÖ Create a repository for your project
  • ‚úÖ Upload files directly in the browser
  • ‚úÖ Only use cloning if you want a local copy
  • ‚úÖ Use branches if you need a safe test version

That's really all you need to start!

Pro Tips for Beginners

Commit Often

Make small, frequent commits rather than large, infrequent ones. This makes it easier to track changes and revert if needed.

Write Clear Commit Messages

Use descriptive messages like "Add user authentication feature" instead of "fix stuff".

Use Branches for New Features

Create a new branch for each feature or bug fix to keep your main code clean.

Always Pull Before Pushing

Run git pull first to avoid conflicts with others' changes.

Common Issues & Solutions

‚ùå "Permission denied"

Solution: Set up SSH keys or use a personal access token for authentication.

‚ùå "Merge conflict"

Solution: Open the conflicted files, resolve differences, then commit the resolution.

‚ùå "Branch is behind"

Solution: Run git pull origin main to get the latest changes.

‚ùå "Can't push to main"

Solution: Create a pull request instead of pushing directly to main branch.

Cursor Rules

What are Cursor Rules?

Cursor Rules are custom instructions that help the AI understand your coding preferences, project structure, and specific requirements. They ensure consistent suggestions across your project.

Setting Up Cursor Rules

1. Create .cursorrules file

Create a .cursorrules file in your project root

2. Define Your Rules

.cursorrules 
# Rules

## Code Style
- Use Python 3.9+ syntax
- Follow PEP 8 style guide
- Use type hints for function parameters and return values
- Use descriptive variable names

## Data Analysis
- Always use pandas for data manipulation
- Include error handling for file operations
- Document data transformations with comments
- Use matplotlib/seaborn for visualizations

## Jupyter Notebooks
- Use markdown cells to explain analysis steps
- Clear output before committing notebooks
- Include data source information in first cell

## File Structure
- Store data files in /data directory
- Save outputs in /outputs directory
- Keep notebooks in /notebooks directory

üîç More Technical Tips

Use "Think First" Prompts

Before asking for code, start with: "Think step-by-step before coding..."

This primes Cursor (especially Claude) to reason before generating ‚Äî reducing hallucinations and improving quality.

Leverage Codebase Search Effectively

Use @codebase or press Cmd+X (Mac) / Ctrl+P (Windows) to search across all project files.

This helps Cursor locate definitions quickly and lets you give better file references like @utils/helpers.py

Use "Explain + Improve" Loops

For tricky code, use a 2-pass approach:

  1. Ask Cursor: "Explain this block"
  2. Then: "How would you improve it?"

This gets cleaner, more readable, and optimized code ‚Äî especially useful when auditing legacy code.

Use YOLO Mode for Prototyping

Great for quick drafts or UI scaffolding.

  • Combine with Git branches so you can roll back safely
  • Always ask it to "summarize what you changed" after using YOLO

Chain Multiple Prompts in Composer

Cursor's Composer lets you run multi-step tasks more reliably:

# Goal: Clean and visualize sales data
1. Load CSV
2. Clean NaN and fix date formats
3. Plot monthly revenue using seaborn

This results in better-structured, testable code than asking for everything in one chat.

Maintain a Prompt Library

Create a prompts/ folder with your reusable:

  • .prompt files (naming, testing, logging prompts)
  • Snippets for common tasks

You'll move faster once you build a personal prompt toolkit.

Reset Context Frequently

If outputs start to degrade:

  • Start a new chat or composer
  • Re-establish project context with a few key files/rules

Long-running chats cause AI to forget earlier logic or make erratic assumptions.

Use Scratchpad for Complex Tasks

Claude models love to think out loud. Try:

"Let's use a scratchpad to work through this logic before writing code."

This helps it design algorithms or class structures better before committing to syntax.

Use Cursor as a Mentor, Not Just Assistant

Instead of only asking for code:

  • Ask: "What are the trade-offs of doing X vs Y?"
  • Or: "What would a senior engineer watch out for in this implementation?"

Customize Workspace Settings

  • Set default models for different file types
  • Enable auto-save, diff-on-save, and integrate your favorite Git tools
  • Tweak theme, keybindings, AI verbosity to match your working style

Common Requests ‚Üí Better Prompts

"Explain this function"
Better:

"Explain what this function does, including its inputs, outputs, and any edge cases it handles."

Why: Makes the request more structured and encourages a thorough response.
"Add error handling"
Better:

"Add robust error handling for invalid inputs, timeouts, and API failures. Use try-except blocks or validation checks where appropriate."

Why: Specifies what kinds of errors to handle, which leads to smarter, more resilient code.
"Optimize this code"
Better:

"Optimize this code for speed and memory usage without changing its functionality. If possible, suggest faster algorithms or more efficient libraries."

Why: Clarifies what "optimize" means and sets boundaries.
"Add documentation"
Better:

"Add clear and concise docstrings for each function and class. Follow the Google Python docstring format and include parameter types and return values."

Why: Guides the style and structure of the documentation.
"Fix this bug"
Better:

"Identify and fix the bug causing [describe the issue briefly, e.g., 'index out of range error when input is empty']. Include a brief explanation of the root cause."

Why: Helps focus the AI (or person) on reproducing and diagnosing the actual issue, not just guessing.
